In 15 seconds: Kano sorts features into 5 types based on how users feel about them. Must-Haves cause anger if missing. Performance features scale linearly — more is better. Delighters surprise and wow. Knowing the type tells you what to build, what to invest in, and what to skip.

🧩

The 5 Categories

Each type affects satisfaction differently

🔴

Must-Haves (Basics)

Expected · Anger if missing

Features users take for granted. They don't increase satisfaction when present — but cause fury when missing. You get zero credit for having them, but you get punished for lacking them.

Hotel: clean towels, working wifi. App: login works, pages load fast.
📈

Performance (Linear)

More = better · Satisfaction scales

The "more is better" features. Battery life, storage space, delivery speed. Satisfaction goes up proportionally with investment. These are what users consciously compare.

Hotel: room size, breakfast quality. App: speed, storage, search accuracy.
✨

Delighters (Attractive)

Unexpected · Wow if present

Surprises users didn't know they wanted. They don't cause dissatisfaction when absent (users never expected them), but create disproportionate joy when present. The "wow factor."

Hotel: handwritten welcome note. App: smart autocomplete, playful animations.
😐

Indifferent

Nobody cares · No impact either way

Features that don't affect satisfaction whether present or absent. Building these is pure waste — they consume resources but move no needle.

Hotel: brand of soap dispenser. App: internal code refactor users never see.
🔄

Reverse

Polarizing · Some users actively dislike

Features that a segment of users actively doesn't want. Adding them reduces satisfaction for that group. Often appears in "simplicity vs power" tradeoffs.

Hotel: loud background music. App: forced social features, gamification.

🧠 Memory trick

Think of a hotel stay: Must-Have = clean room (rage if dirty). Performance = room size (bigger is better). Delighter = surprise champagne (wow!). Indifferent = brand of hangers. Reverse = loud music in lobby.

The Satisfaction Curve

How each category behaves as you invest more

🔍 Reading the curves

Must-Haves (red) — starts deep negative (anger), rises to neutral, but never above. You can't delight with basics — you can only stop disappointing.

Performance (blue) — straight diagonal. More investment = proportionally more satisfaction. This is where competitive features live.

Delighters (purple) — flat at first (no one expects them), then rockets upward. Small investments here can create outsized emotional responses.

⏳ Categories drift over time

What's a Delighter today becomes a Performance feature tomorrow and a Must-Have next year. Example: smartphone GPS was a Delighter in 2008, Performance by 2012, Must-Have by 2015. Re-evaluate your Kano categories at least annually.

The Survey Method

Two questions per feature — that's the whole trick

The Kano Questionnaire

For each feature, ask users TWO questions:

Functional (positive)

"If this feature IS present, how do you feel?"

😍 I like it
✅ I expect it
😐 I'm neutral
🤷 I can tolerate it
😤 I dislike it
Dysfunctional (negative)

"If this feature is NOT present, how do you feel?"

😍 I like it
✅ I expect it
😐 I'm neutral
🤷 I can tolerate it
😤 I dislike it

📋 The prioritization takeaway

Build order: 1) Fix any broken Must-Haves first (non-negotiable). 2) Invest in Performance features (competitive advantage). 3) Sprinkle in 1–2 Delighters (differentiation). 4) Cut Indifferent and Reverse features.

When to Use

Powerful for product strategy, overkill for sprint planning

✅ Use Kano when...

  • Deciding product strategy — what TYPE of features to invest in
  • You can survey 30+ real users (need data for reliability)
  • Comparing "table stakes" vs "differentiators"
  • Building a new product and mapping the feature landscape
  • Competitive analysis — understanding where rivals invest

🚫 Skip Kano when...

  • You need to rank 15 features for next sprint → use RICE
  • No access to real users for the survey
  • Time-sensitive decision (Kano surveys take 1–2 weeks)
  • Internal tooling where you ARE the user
  • Very early MVP — just ship and learn from real usage

🔗 How Kano connects to other frameworks

Kano → tells you WHAT TYPE to build (Must-Haves first, then Performance, then Delighters). RICE → tells you WHICH ONES within a type. Value vs Effort → quick first-pass sort. Use all three together for a complete picture.

⚠️

6 Common Pitfalls

Kano is subtle — these mistakes ruin the results

1

Skipping the survey and guessing

PMs often classify features based on gut feel. But YOU are not the user. Your "Delighter" might be users' "Indifferent."

Run the actual paired survey with 30+ users. The whole point of Kano is user data, not PM intuition.
2

Treating categories as permanent

Features drift: today's Delighter → tomorrow's Must-Have. GPS, touchscreens, dark mode all followed this path.

Re-run Kano analysis annually. What delighted users last year may be expected now.
3

Ignoring Must-Haves because they're "boring"

Teams love building Delighters. But if your Must-Haves are broken, Delighters won't save you — users will leave anyway.

Always audit Must-Haves first. Only invest in Delighters once basics are solid.
4

Confusing the survey questions

Users find the dysfunctional question ("how do you feel if NOT present?") confusing. Bad survey → bad data → wrong categories.

Use plain language. Test the survey with 3 people first. Consider rephrasing to "imagine this feature didn't exist."
5

Too few respondents

With 5 responses, one outlier changes the category. You need enough data for the results to be reliable.

Minimum 30 responses for directional results, 100+ for confidence. Segment by user type if possible.
6

Trying to use Kano for ranking

Kano tells you the TYPE, not the ORDER. "Both are Must-Haves" doesn't tell you which to build first.

Use Kano for categorization, then RICE or Value vs Effort to rank within each category.

Run a Session

2-phase process: Survey (async) + Workshop (60 min)

📧 Phase 1: Survey (Async, 1–2 weeks)

Write the Kano questionnaire (2 questions per feature × 8–12 features). Send to 30–100 users via Typeform, Google Forms, or similar. Analyze results using the evaluation table to get majority categories.

Keep to 8–12 features max. More than that and survey fatigue kills your completion rate.

👥 Phase 2: Workshop

Must have

PM, Design lead, Eng lead, whoever ran the survey

Nice to have

UX researcher, product marketing, customer support lead

0–10

Present the results

Show the Kano categories for each feature. Share the raw numbers — "72% said Must-Have, 18% Performance."

10–25

Discuss surprises

Focus on features where the team assumed one category but users said another. These are the gold — your blind spots revealed.

25–40

Audit Must-Haves

Are any broken or missing? These are non-negotiable — fix them before touching anything else. Create action items.

40–50

Plan Performance investments

Which Performance features will differentiate you? How much should you invest? These are your competitive levers.

50–60

Pick Delighter bets

Choose 1–2 Delighters to prototype. These are high-risk, high-reward — validate cheaply before committing.

💡 Facilitation tips

Show raw percentages — "72% Must-Have" is more convincing than "we classified this as Must-Have." Data shuts down opinion debates.

Focus on disagreements — when survey says Delighter but the PM assumed Must-Have, that's where the best learning happens.

Don't rank in this session — Kano categorizes. Use a separate RICE session to rank within categories.

Quick Quiz

6 questions to lock it in

1. A feature users expect but never praise — they only notice when it's MISSING. What is it?

Must-Haves = expected, invisible when present, infuriating when absent. Like a hotel having hot water.

2. User answers: "I like it" if present, "I'm neutral" if absent. What category?

Like if present + neutral if absent = Delighter. They enjoy it but don't miss it — classic unexpected delight.

3. Your team wants to build a flashy new feature, but login is broken 10% of the time. What first?

Always fix broken Must-Haves first. Delighters can't overcome broken basics — users will leave before experiencing the wow.

4. What happens to Kano categories over time?

Categories drift downward: Delighter → Performance → Must-Have. Today's wow is tomorrow's expectation. Re-evaluate annually.

5. How many survey responses do you need for reliable Kano results?

Minimum 30 for directional results. With fewer, one outlier can change the entire category.

6. Kano tells you the category, but not the order. What do you use to RANK features within a category?

Kano categorizes. RICE ranks. They're complementary — use Kano to decide what TYPE, then RICE to decide which ones within that type.
