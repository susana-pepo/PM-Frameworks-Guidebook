📥 Input / Output Metrics

Separate what you can control (inputs) from what you're trying to achieve (outputs). Then focus your team on the levers that actually move.

⚡

One-liner: Output metrics = business results you want (revenue, retention, growth). Input metrics = actions your team directly controls that drive those outputs. Focus teams on inputs, track progress on outputs.

The Core Idea

Control what you can, measure what matters

The Problem: Teams Chasing Outputs

A team's OKR says "Increase D30 retention from 8% to 15%." Great goal. But what do they do on Monday morning? Retention is an output — you can't directly change it. You can only change things that influence it.

The Input/Output model separates what you control (inputs: ship features, send emails, improve onboarding) from what you're trying to achieve (outputs: retention, revenue, growth). The magic: when you connect the two clearly, teams know exactly what to do every day.

🎯 Input Metrics
Things you directly control. Actions, features, experiments. "We shipped 5 new lessons." "We sent re-engagement emails on Day 3." "We reduced onboarding to 3 steps." These are leading indicators.
📊 Output Metrics
Business results you want to achieve. Retention, revenue, NPS, growth rate. You can't directly change these — they're the result of your inputs. These are lagging indicators.

🧠 Memory Trick

Think of a vending machine. The buttons you press = inputs (you control them). What comes out = outputs (result of your choices). You can't reach in and grab the snack directly — you have to press the right buttons. Same with metrics: you can't "do retention." You can only do things that cause retention.

🔍 Input Metric Qualities

✅
Controllable
Your team can directly move it through their work. "Lessons published per week" — yes. "App Store ranking" — no (too many external factors).
✅
Predictive
When this input goes up, the output follows (with some lag). You need to validate this connection with data. "More push notifications" doesn't always mean "more retention."
✅
Timely
Moves faster than the output. Inputs should give you weekly (ideally daily) signal. If you have to wait 6 months to know if an input worked, it's too slow.

🚩 The Correlation Trap

Not every input actually drives the output. "We sent more emails" (input) doesn't always mean "retention improved" (output). You must validate the connection with data or experiments. A/B test: does this input actually move the output? If not, find a better lever.

Deep Dive

Counter-metrics and metric trees

Counter-Metrics: The Safety Net

Every input can be gamed or over-optimized. If your input is "push notifications sent," a team might blast users 10x a day — great for the input metric, terrible for user experience. Counter-metrics catch this.

Input: Push notifications sent → Counter: Unsubscribe rate
If notifications go up but unsubscribes spike, you're hurting users.
Input: New features shipped → Counter: Bug count, support tickets
If features go up but quality drops, you're creating tech debt.
Input: Free trial signups → Counter: Trial-to-paid conversion rate
If signups go up but conversion drops, you're attracting low-quality users.

🌳 The Metric Tree

A metric tree decomposes your output into a hierarchy of inputs. Each branch represents a different lever. This makes it visual and actionable.

💡 The Decomposition Rule

Any output can be decomposed into inputs using math: Revenue = Users × Conversion × ARPU. Each factor becomes a team's input metric. This is how you go from "increase revenue" (vague) to "improve trial-to-paid conversion from 3% to 5%" (actionable).

🚩 Real-World Tension

The Content team publishes 5 new lessons (input up!), but lesson completion rate drops (counter-metric down). What happened? New lessons are too hard. The counter-metric caught the problem. Without counter-metrics, you'd celebrate the input and miss that you're hurting the output.

When to Use

Right situations and wrong ones

✅ Use Input/Output When...

  • ✔️ Setting team-level OKRs and goals
  • ✔️ Operationalizing your North Star
  • ✔️ PM interview asks "how would you measure success?"
  • ✔️ Building a product analytics dashboard
  • ✔️ Cross-functional alignment — who owns what metric
  • ✔️ Sprint planning — what inputs are we moving this sprint?

⛔ Skip When...

  • ❌ You haven't defined your North Star yet (do that first)
  • ❌ Pre-product-market-fit (too early to optimize)
  • ❌ Diagnosing funnel problems (use AARRR)
  • ❌ High-level strategic direction (use Playing to Win)
  • ❌ Feature-level prioritization (use RICE)

🔗 Connections

Growth Loops
Each step in a growth loop has its own input/output pair. Optimize each step.

🎯 The Complete Growth Stack

AARRR (diagnose funnel) → North Star (pick THE metric) → Input/Output (decompose into team levers) → Growth Loops (design compounding systems) → OKRs (quarterly execution). This is the full Growth → Execution pipeline.
